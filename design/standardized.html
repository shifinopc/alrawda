This addendum standardizes two areas requested for Phase 1. It sits alongside FLOW-RAWDA-ERP-003 — Complete Menus & Flows; the receipt-approval step replaces the informal “receipt = approval” behaviour with a clear, auditable approval workflow, and adds the User Management menu.

1. Receipt Approval — Overview

In day-to-day work, a customer often takes several packages at once (for example an Umrah package plus extra transport or an additional hotel night). The counter staff record the customer’s payment as a receipt. At the end of the day, staff select the receipts they have created and send them to the manager with the total amount; the manager reviews and approves them. Only approved receipts are treated as confirmed income.

This document standardizes that process so every receipt follows the same path, the manager has one place to approve from, and the audit trail is complete.

2. One Customer, Several Packages

How a multi-package booking is captured

Standard rule: a single booking/invoice for the customer can hold several package line items. The receipt is raised against that booking and can cover one or all of the packages. This keeps one customer’s money in one place and makes the balance and approval straightforward.

  • Each package is a line item on the booking (package, qty/pax, rate, line total). The booking net is the sum of lines.
  • A receipt records the amount received and reduces the booking balance regardless of how many packages it contains.
  • The printed receipt voucher lists the packages covered, the amount paid, and the remaining balance — matching today’s bilingual voucher.
If the agency prefers, several separate invoices for the same customer can instead be grouped under the customer and a single receipt applied across them. The standardized rule is simply: one receipt, clearly linked to the customer’s booking(s), with a correct balance.

3. Receipt → Manager Approval Flow

Standardized — counter staff & manager

Counter staff Manager System

  1. Create receipts. Counter staff record each customer’s payment as a receipt (covering one or several packages). Each new receipt is Draft.
  2. Select receipts. On the Receipt Approval screen, staff tick the receipts to submit (typically the day’s collection).
  3. Submit to manager. Staff submit the selected set; the system shows the number of receipts and the total amount and marks them Submitted (pending approval).
  4. Manager reviews. The manager opens the approval queue, sees the selected receipts and total, and checks each (customer, packages, amount, mode).
  5. Approve / reject. Approved receipts become Approved / Posted and lock; rejected ones return to staff with a comment to correct and resubmit.
  6. Reports update. Only approved receipts count as confirmed income in the reports.
What changes vs today. Instead of the receipt itself being the informal approval, the manager now formally approves a selected batch of receipts with its total — exactly the daily routine described — and the system records who approved what and when.

5. User Management — Concepts & Flow

Settings → User Management

Access is controlled by roles. A role is a named set of permissions defined per menu and per action (View, Create, Edit, Delete, Print, Approve). Each user is given a role; the user then sees and can use only what the role allows.

  1. Roles: create/edit roles and set their menu-wise, action-wise permissions.
  2. Users: create a user (name, login, email/mobile, password), assign a role, set active/inactive, optional 2FA.
  3. Login: the user sees only permitted menus; blocked actions are hidden or disabled.
  4. Maintenance: reset password, deactivate on exit, reassign role; every change is audited.

7. Sample Roles & Enforcement

7.1 Sample roles

RolePurpose
AdministratorFull system access including User Management, master data, settings.
ManagerApproves/rejects receipts, sees all reports, manages bookings; no user administration.
Counter staffCreates invoices and receipts, prints vouchers, submits receipts for approval. Cannot approve, delete or see financial settings.
AccountantHandles payments, expenses, refunds and financial reports.
Viewer / AuditorRead-only access to records and reports.

7.2 How permissions are enforced

  • Menu visibility: a user only sees menu items their role permits — the rest are hidden.
  • Action control: buttons (New, Edit, Delete, Print, Submit, Approve) are shown only if allowed.
  • Server-side check: every request is also checked on the server, so permissions cannot be bypassed.
  • Separation of duties: the person who submits a receipt cannot be the one who approves it — Approve is manager-only.
  • Audit log: logins, permission changes and approvals are all recorded.
