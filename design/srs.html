1. Introduction & Purpose

AL RAWDA GROUP HAJJ & UMRAH SERVICES currently operates a Windows desktop application (“Stimes ERP Solution”) to manage its Umrah and Hajj travel business — issuing invoices, collecting payments, recording office expenses, maintaining packages and visa types, and printing bilingual receipts and management reports.

The business wants to move to a modern, cloud-hosted application built on a MySQL database, accessible from any device, with stronger security, AI-assisted automation, and the flexibility to add or change form fields and printed documents without a code release. This Software Requirements Specification (SRS) defines what the new system must do and the technical approach ionob proposes to build it.

1.1 Objectives of this document

  • Capture and confirm the functional and non-functional requirements of the new platform.
  • Document the proposed architecture, technology stack and data model.
  • Define the dynamic field and dynamic print engines that are central to the client’s request.
  • Establish the security model, AI scope, migration approach and delivery plan.
  • Serve as the agreed baseline for design, build, testing and sign-off.

1.2 Intended audience

The client’s management and operations team (for scope confirmation), and ionob’s engineering, QA and project teams (for design and build).

3. Project Scope & Objectives

3.1 Business objectives

  • Operate the entire Umrah/Hajj sales-to-collection cycle from a secure cloud application accessible on desktop, tablet and mobile.
  • Let authorised staff configure form fields and printed documents themselves, without waiting for a software release.
  • Reduce manual effort and errors through AI-assisted automation.
  • Strengthen security, access control and auditability of all financial records.
  • Preserve the familiar bilingual workflow and the existing voucher/report formats.

3.2 In scope

Functional

Master data, Invoicing, Receipts, Payments (expense & refund), Approval workflow, full Report set, Dashboards, Dynamic field engine, Dynamic print engine, User & role management, Audit log, AI assistant features.

Platform

Cloud hosting, MySQL database, REST API, responsive React web client, bilingual EN/AR UI with RTL, role-based security, automated backups, data migration from the current system.

3.3 Out of scope (this phase)

  • Native mobile apps on app stores (the responsive web app covers mobile; native apps can be a later phase). TBC
  • Direct integration with airline GDS / visa-issuing government portals unless APIs are provided. TBC
  • Full double-entry general-ledger accounting (the system covers operational income/expense; export to an accounting package can be added). TBC
  • Payroll/HR beyond recording salary payments as expenses.

3.4 Success criteria

  • All current daily operations can be performed in the new system, faster and from anywhere.
  • Existing vouchers and reports are reproduced to the client’s satisfaction, and can be edited by an admin without code.
  • Historical data is migrated accurately and reconciled with the old system’s totals.
  • The system passes security testing and the agreed acceptance tests.

4. Proposed Solution & Architecture

The platform is a cloud-hosted, multi-tenant-ready web application with a clear separation between the React front-end, a Node.js REST/JSON API, and a managed MySQL database. All access is over HTTPS and protected by token-based authentication and role-based authorisation.

4.1 Architectural principles

  • API-first. Every screen and report is powered by a documented REST API, enabling future mobile apps and integrations.
  • Stateless app tier. Application servers hold no session state, so the system scales horizontally behind a load balancer.
  • Separation of concerns. Presentation (React), business logic (Node API), and data (MySQL) are independent layers.
  • Configuration over code. Fields, documents, packages, visa types and roles are data, not code — supporting the dynamic requirements.
  • Security by default. Encryption in transit and at rest, least-privilege access, and full audit logging.
  • Multi-tenant ready. Schema supports multiple branches/companies with isolated data, should the group expand.

7. Dynamic Field Engine

A core client requirement is that fields can be added or changed without a code release. The dynamic field engine lets an authorised administrator define custom fields on any entity (invoice, receipt, payment, customer, passenger, package) through a configuration screen.

7.1 Capabilities

  • Add fields of types: text, number, currency, date, dropdown (from a managed list or master data), checkbox, multi-select, phone, email, file attachment, and computed/formula fields.
  • Set label (English + Arabic), help text, default value, placeholder, required/optional, min/max, regex validation, and display order/section.
  • Control visibility and edit-ability by role, and conditional display (“show field B when field A = X”).
  • Mark a field as printable and map it into document templates (Section 8).
  • Activate/deactivate or version a field without losing historical data.

7.2 How it is stored

Custom field definitions live in metadata tables; custom field values are stored per record. The engine uses MySQL JSON columns plus a definitions table, giving flexibility without schema migrations while keeping core columns strongly typed for reporting.

TablePurpose
field_definitionsEntity, key, type, labels (EN/AR), validation, options, role visibility, print flag, order, status
<entity>.custom_data (JSON)Key→value store of custom field values for each record
field_auditWho changed which definition and when

8. Dynamic Print & Document Engine

The second core requirement is dynamic printing: every printed document (invoice, receipt voucher, payment voucher, reports) is driven by an editable template rather than hard-coded layout. Templates render to pixel-accurate PDF via a headless-Chromium HTML/CSS pipeline, fully supporting bilingual English/Arabic content and right-to-left layout.

8.1 Capabilities

  • WYSIWYG / HTML template editor with company header, logo, address and Arabic/English blocks (matching the current vouchers).
  • Insert any standard or dynamic field as a placeholder (e.g. {{customer.name}}, {{invoice.net_amount}}, {{custom.passport_no}}).
  • Repeating sections (passenger lists, line items) and conditional blocks (show terms only on receipts).
  • Per-document settings: page size, margins, header/footer, watermark, language(s), and signature lines.
  • Multiple templates per document type with versioning; set a default and preview before publishing.
  • One-click reprint, bulk print, email as PDF, and download.

10. Security & Compliance

10.1 Authentication & access

  • Individual user accounts with strong password policy; passwords hashed with argon2/bcrypt.
  • JWT access/refresh tokens; automatic session expiry and refresh.
  • Optional two-factor authentication (TOTP) for admin and finance roles.
  • Role-Based Access Control (RBAC) down to module and action level.
  • Account lockout after repeated failed logins; forced password reset flows.

10.2 Roles (initial)

RoleTypical permissions
AdministratorFull access incl. users, roles, dynamic fields, templates, settings
ManagerApprove invoices, view all reports & dashboards, void/refund with reason
Counter staff / AgentCreate invoices, receipts, payments; print; no deletions or approvals
AccountantPayments, expenses, refunds, financial reports & exports
Viewer / AuditorRead-only access to records, reports and audit log

Roles and permissions are configurable; the above is a starting point for confirmation.

10.3 Data protection

  • TLS 1.2+ for all traffic (HTTPS only); HSTS enabled.
  • Encryption at rest for the database, backups and document storage.
  • Tenant/branch data isolation; row-level scoping by branch where applicable.
  • Daily automated backups with retention and tested restore; point-in-time recovery.
  • Secrets stored in a managed vault, never in code.

10.4 Auditability & hardening

  • Full audit trail: who created/edited/deleted/approved which record and when (immutable log).
  • Soft-delete with reason for financial records; no silent hard deletes.
  • Protection against OWASP Top 10 (input validation, parameterised queries, output encoding, CSRF/XSS/SQLi defences).
  • Web Application Firewall, rate limiting and brute-force protection at the edge.
  • Dependency scanning and a security review before go-live.

13. Data Migration Strategy

Historical data from the current desktop application is migrated into the new MySQL database so the business has a single, continuous record.

  1. Assess source. Identify the current database engine and export options (direct DB access, backup file, or report/CSV export). TBC
  2. Map data. Map source tables/fields to the new schema (customers, packages, visa types, nationalities, invoices, passengers, receipts, payments).
  3. Build ETL. Scripts to extract, clean (de-duplicate, fix encoding incl. Arabic), transform and load, preserving original document numbers.
  4. Trial migration. Load into staging; reconcile counts and financial totals against the old system’s reports.
  5. Validation & sign-off. Client verifies key records and report totals match.
  6. Cutover. Final delta migration, freeze old system, go live; keep the old system read-only as a fallback for an agreed period.
Dependency. Migration scope and effort depend on access to the current data. Earliest possible sharing of a database export or sample helps confirm the plan.
