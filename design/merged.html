1. Scope & How to Read

This is Phase 1: rebuilding the agency’s existing application as a cloud system on MySQL — same menus, same screens, same flows the staff already know — with two additions the business asked for: a proper receipt → manager approval flow and user management with role and menu-wise permissions. Larger enhancements (enquiry/quotation, the Nusuk hotel→visa chain, dashboards) are planned for a later phase.

For every menu you get a short purpose, a flow diagram, the steps, and the screen fields. Required fields are marked *.

Menu / user action System / output Decision Screen / form

Every transaction and master screen uses a List + Detail layout with New, Save, Delete and search/filter. Menus open as tabs so several records stay open at once — matching the current application.

8. Receipt Approval (Receipt → Manager)

Transaction → Activity → Receipt Approval

The standardized approval the business uses: counter staff select the day’s receipts and submit them to the manager with the total amount; the manager reviews and approves. Only approved receipts count as confirmed income.

Counter staffManagerSystem

  1. Staff create receipts (each Draft), then select the ones to submit (the day’s collection).
  2. Staff submit; the system shows the receipt count and total amount and marks them Submitted.
  3. The manager opens the approval queue, checks each receipt, and approves (→ Approved/Posted, locked) or rejects (returns to staff with a comment).
  4. Only approved receipts count in the income reports.
RuleBehaviour
EditableOnly while Draft; locked after approval (changes need void + re-entry, audited)
IncomeOnly approved receipts count in Income reports
Batch totalSubmission shows the total for the manager to confirm against cash/bank received
Separation of dutiesSubmit = staff; Approve/Reject = manager only
AuditSubmit, approve, reject all record user, time, comment

11. User Management & Permissions

Settings → User Management

Access is controlled by roles. A role defines permissions per menu and per action (View, Create, Edit, Delete, Print, Submit, Approve). Each user gets a role and then sees only what it allows.

Enforcement

  • Menus and buttons appear only if the role permits; every request is also checked on the server so it can’t be bypassed.
  • Separation of duties: whoever submits a receipt cannot approve it — Approve is manager-only.
  • User fields: name, login, email/mobile, password, role, active/inactive, optional 2FA. Logins and permission changes are audited.

Putting it together — end to end
Customer: Mohammad Jahirul Islam (Bangladeshi), family of 5 passengers, “5 Days Umrah With Visa”, departing 10/06/2026. Total QAR 6,450, paid in two installments. (Based on Invoice 8457.)
StageScreen & entryPrintedBalance
1. VisitTake name, nationality, 5 passenger names, package, departure——
2. InvoiceInvoice → New; package line + 5 passengers; Net 6,450; SaveInvoice (bilingual)Due 6,450
3. Receipt #1Receipt → New → select Invoice 8457; Cash 5,000; SaveReceipt Voucher #11,450
4. ApprovalStaff submit the day’s receipts; manager approves—1,450
5. Receipt #2Customer returns; Receipt → select 8457; 1,450; Save & approvedReceipt Voucher #20 (settled)

On departure (10/06/2026) the 5 passengers appear on the Passengers List. If a passenger cancels, a Payment → Refund linked to the invoice adjusts the income.
