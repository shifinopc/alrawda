1. Scope & How to Read

This document covers Phase 1: rebuilding the agency’s existing application as a cloud system that behaves the same way staff already know — same menus, same screens, same flows — on a MySQL database. Operational enhancements (lead/quotation, Nusuk hotel→visa chain, formal approval gates, dashboards) are deliberately out of scope here and planned for a later phase.

For every menu you will find: a short purpose, a flow diagram, the numbered steps, and a field reference (the fields on that screen). Required fields are marked *.

Menu / user action System response / output Decision Screen / form

Every transaction and master screen uses the same layout as today: a List on the left and a Detail form on the right with New, Save and Delete actions, plus search/filter on list columns. All menus open as tabs across the top bar so several records stay open at once.

3. Overall Transaction Flow

End-to-end — how the current model is used day to day

Set up master data once, then run the daily cycle: raise an Invoice, collect money with a Receipt (which also serves as the approval/confirmation document), record office Payments for expenses and refunds, and read it all back through Reports.

4. Login & Main Navigation

Entry point
  1. Open the application in a browser and log in (username, password).
  2. The system validates and opens the main menu; the user has the top menu (Transaction, Settings) and can open any item as a tab.

5. Package

Transaction → Basic → Package

Maintains travel packages and their rates (e.g. “5 Days Umrah With Visa — 1050”).

  1. Open Package; the list loads with search/filter.
  2. Click New, or select a row to edit.
  3. Enter the package name and rate, then Save. Use Delete to remove/deactivate.

Fields

8. Invoice

Transaction → Activity → Invoice

Creates the sale: customer, package, passengers and amounts.

  1. Open Invoice; click New. An invoice number and date are generated.
  2. Fill the header, add passenger rows; the system computes amount, discount and net amount.
  3. Save, then Print the bilingual invoice and proceed to a Receipt for collection.

9. Receipt

Transaction → Activity → Receipt

Collects money against an invoice and prints the bilingual Receipt Voucher (سند قبض). In the current model the receipt also serves as the booking confirmation / approval document.

  1. Open Receipt, click New, and select the invoice; details auto-fill.
  2. The system shows invoice amount and previous balance.
  3. Enter the received amount and mode of payment, Save, then Print the voucher; the balance updates. Repeat for further installments.

10. Payment

Transaction → Activity → Payment

Records office expenses and customer refunds.

  1. Open Payment, click New, choose Type: Expense or Refund.
  2. Enter paid-to, mobile, narration and amount (e.g. office, salary, taxi, printing for expenses).
  3. Save and Print the voucher; expenses feed the Expense report.

11. All Report

Transaction → Reports → All Report

One report screen produces all six report types by selection and date range.

  1. Open All Report and choose the report type (radio button).
  2. Set Start Date and End Date, then Print to preview and export.

Report types

ReportShows
Income SummaryInvoice / received / refund / income / balance per invoice, with totals
PendingInvoices with outstanding balance
Income DetailDetailed income breakdown
Passengers ListPassengers grouped by departure date with invoice & customer
Departure Date wiseInvoices grouped/filtered by departure with approval status
ExpensePayment vouchers with narration and grand total

12. Settings

Top bar → Settings

Administrator area for company details, users and basic system options.

  1. Open Settings (admin only).
  2. Maintain company info & branding (used on all vouchers), users & roles, and basic print/field options; Save.
Advanced dynamic-field and template editing, dashboards and the Nusuk operational chain are planned for a later phase; Phase 1 keeps Settings to the essentials the current app needs.

13. Common Screen Behaviour & Field Key

13.1 Standard actions on every screen

ActionBehaviour
List + DetailList of records on the left; selected record opens in the detail form on the right
Search / FilterInline filter on list columns; sort and paginate
NewClears the form for a new record; auto number & date where applicable
SaveValidates required fields, saves, refreshes the list
DeleteRemoves/deactivates; blocked if referenced by other records
PrintGenerates the bilingual PDF (invoice, receipt, payment, reports)
TabsEach menu opens as a tab; multiple records open at once

13.2 Field-type key

TypeMeaning
AutoGenerated by the system (numbers, dates, carried-over values)
Text (EN/AR)Free text; bilingual where shown
NumberNumeric / currency (QAR)
DateCalendar date

OptionRadio / choice (e.g. Cash/Bank, Expense/Refund, Room Type)
TableGrid of rows (passenger details)
*Required field

14. Worked Example — One Customer’s Journey

Putting it together — a single customer, end to end

This follows one real-style customer through the current-model system, screen by screen, so the team can see exactly how a booking is managed and how the balance is kept correct from first visit to final report.

Our customer: Mohammad Jahirul Islam (Bangladeshi), booking a family of 5 passengers on the “5 Days Umrah With Visa” package, departing 10/06/2026. Total QAR 6,450. He pays in two installments. (Based on Invoice 8457 in the sample data.)

14.1 The journey at a glance

14.2 Step by step — what staff do at each screen

StageScreen & entryWhat is printedBalance after
1. Visit / enquiryFront desk takes details: name, nationality (Bangladeshi), mobile, the 5 passenger names, package & departure date——
2. InvoiceInvoice → New. Header filled, 5 passenger rows added (name + visa type). System computes Net = 6,450. Save.Invoice (bilingual)Due 6,450
3. Receipt #1Receipt → New → select Invoice 8457. Details auto-fill; shows invoice 6,450, previous balance 6,450. Mode = Cash, Received = 5,000. Save.Receipt Voucher #1 (سند قبض)1,450
4. Receipt #2Customer returns Day 5. Receipt → New → select Invoice 8457. Previous balance now 1,450. Received = 1,450. Save.Receipt Voucher #20 (settled)
5. DepartureOn 10/06/2026 the customer’s 5 passengers appear on the Passengers List for that date.Passengers List / manifest0

If a passenger cancels, staff raise a Payment → Refund linked to the invoice; the refund amount reduces the recorded income and is shown on the Income Summary.

14.3 Where this customer appears in the reports

ReportThis customer shows as…
Income SummaryInvoice 6,450 · Received 6,450 · Refund 0 · Income 6,450 · Balance 0
PendingListed with balance 1,450 after Receipt #1; disappears once Receipt #2 settles it
Income DetailBoth receipts (5,000 + 1,450) itemised against Invoice 8457
Passengers ListAll 5 passengers under departure 10/06/2026
Departure Date wiseInvoice 8457 grouped under 10/06/2026 with its approval status
ExpenseNot shown (no expense/refund raised for this customer)
The key control: selecting the invoice on every receipt is what keeps one customer’s payments tied to one booking, so the balance is always correct and the Pending report is trustworthy — the heart of how a customer is managed in the current model.
