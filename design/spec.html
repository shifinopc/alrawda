1. Introduction & Purpose

This document specifies the cloud-based Umrah & Hajj travel ERP that replaces the current Windows desktop application (“Stimes ERP Solution”). It defines what the system must do (functional specification), how it is structured (technical specification), and how work moves through it (process flows). It is the reference for design, build and testing.

1.1 Scope of the system

The system runs the complete Umrah/Hajj sales-to-collection cycle: maintaining master data, issuing invoices, collecting payments through receipts, recording expenses and refunds, approving transactions, and producing bilingual vouchers and management reports. It is delivered as a secure cloud application on a MySQL database, accessible from desktop, tablet and mobile, with a dynamic field engine, a dynamic print engine, and AI-assisted automation.

1.2 How requirements are written

Functional requirements use the identifier FR-<module>-<n>; non-functional use NFR-<n>. Priority is shown as M Must, S Should, C Could. Items needing a client decision are marked TBC.

3. Architecture & Technology

A cloud-hosted web application with a clear separation between the React front-end, a Node.js REST API, and a managed MySQL database. The app tier is stateless and scales horizontally; all access is over HTTPS with token-based authentication and role-based authorisation.

5. Dynamic Field Engine

Fields can be added or changed without a code release. An authorised administrator defines custom fields on any entity (invoice, receipt, payment, customer, passenger, package) through a configuration screen.

5.1 Capabilities

  • Field types: text, number, currency, date, dropdown (managed list or master data), checkbox, multi-select, phone, email, file attachment, computed/formula.
  • Per field: label (EN + AR), help text, default, placeholder, required, min/max, regex validation, display order/section.
  • Role-based visibility and edit-ability; conditional display (“show B when A = X”).
  • Mark a field printable and map it into document templates.
  • Activate/deactivate or version a field without losing historical data.

5.2 Storage model

Field definitions live in metadata tables; field values are stored per record in MySQL JSON columns, giving flexibility without schema migrations while core columns stay strongly typed for reporting.

TablePurpose
field_definitionsEntity, key, type, labels (EN/AR), validation, options, role visibility, print flag, order, status
<entity>.custom_data (JSON)Key→value store of custom field values per record
field_auditWho changed which definition and when

6. Dynamic Print & Document Engine

Every printed document is driven by an editable template, not hard-coded layout. Templates render to pixel-accurate PDF via headless Chromium, fully supporting bilingual English/Arabic content and right-to-left layout.

6.1 Capabilities

  • Template editor with company header, logo, address, EN/AR blocks (matching current vouchers).
  • Insert any standard or dynamic field as a placeholder, e.g. {{customer.name}}, {{invoice.net_amount}}, {{custom.passport_no}}.
  • Repeating sections (passenger lists, line items) and conditional blocks (show terms only on receipts).
  • Per-document settings: page size, margins, header/footer, watermark, language(s), signature lines.
  • Multiple templates per document type with versioning; set default and preview before publishing.
  • One-click reprint, bulk print, email as PDF, download.

6.2 Documents reproduced

DocumentNotes
InvoiceBilingual, passenger grid, amount/discount/net, remarks
Receipt Voucher (سند قبض)Terms text, received-amount box, package, passenger details, total & remaining balance, three signatures
Payment VoucherExpense/refund, paid-to, narration, amount

Bilingual & RTL. The engine handles mixed Latin/Arabic strings, RTL alignment, and Arabic-Indic numerals where required, so new vouchers match the existing ones while remaining admin-editable.

8. Security & Access Control

8.1 Authentication & access

  • Individual accounts, strong password policy, passwords hashed with argon2/bcrypt.
  • JWT access/refresh tokens; automatic session expiry/refresh.
  • Optional 2FA (TOTP) for admin and finance roles.
  • Role-Based Access Control down to module and action level.
  • Account lockout after repeated failed logins; forced password reset.

8.2 Roles (initial)

RoleTypical permissions
AdministratorFull access incl. users, roles, dynamic fields, templates, settings
ManagerApprove invoices, view all reports/dashboards, void/refund with reason
Counter staff / AgentCreate invoices, receipts, payments; print; no deletions/approvals
AccountantPayments, expenses, refunds, financial reports & exports
Viewer / AuditorRead-only access to records, reports and audit log

8.3 Data protection & auditability

  • TLS 1.2+ everywhere (HTTPS only, HSTS); encryption at rest for DB, backups and documents.
  • Tenant/branch data isolation; row-level scoping by branch where applicable.
  • Daily automated backups with tested restore; point-in-time recovery.
  • Full immutable audit trail: who created/edited/deleted/approved what and when.
  • Soft-delete with reason for financial records; OWASP Top 10 defences; WAF, rate limiting, dependency scanning.

12. Data Migration

Historical data from the current desktop application is migrated into the new MySQL database so the business keeps a single, continuous record.

  1. Assess source. Identify the current database engine and export options (direct DB, backup file, or CSV export). TBC
  2. Map data. Map source tables/fields to the new schema (customers, packages, visa types, nationalities, invoices, passengers, receipts, payments).
  3. Build ETL. Extract, clean (de-duplicate, fix Arabic encoding), transform and load, preserving original document numbers.
  4. Trial migration. Load into staging; reconcile counts and financial totals against the old system’s reports.
  5. Validation. Verify key records and report totals match.
  6. Cutover. Final delta migration, freeze old system, go live; keep the old system read-only as a fallback for an agreed period.
Dependency. Migration effort depends on access to the current data. An early database export or sample confirms the mapping.
